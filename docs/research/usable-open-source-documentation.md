# Research: Building Usable Documentation for an Open-Source Project (and Applying It to Han)

## Summary

Build documentation usability in two layers, and treat them as mostly separate decisions. The **baseline layer** is well-supported and low-risk: keep docs in the repository under code review (docs-as-code), organize them by reader need rather than by internal structure, write each page to stand on its own, give every plugin a short "front door" README, and lean on GitHub's native rendering (Mermaid diagrams, auto-generated tables of contents, relative links) instead of a build pipeline. This baseline delivers the structural asks in issue #115 — a README per plugin, slimmer indexes, a plugin index. The **escalation layer** — standing up a rendered documentation site (MkDocs, Sphinx, Docusaurus) on GitHub Pages or Read the Docs — has no clear winner in the evidence and should be deferred until the plugin-centric reorganization is done, then decided against named criteria.

For Han specifically, one question issue #115 raises is **not** settled by this research and must be decided during implementation: how much a per-plugin README says about each skill and agent. Han's existing "one canonical source per concept" rule wants scent-plus-link only, but Han's own draft plugin-README guidance (`docs/plugin-readme.md`) currently specs a fuller per-skill section with paragraph descriptions and example prompts that would duplicate the long-form docs. Those two pull against each other, and issue #115 flags exactly this tension as open. The safe reading of the research is: keep the per-plugin README to scent-plus-link and trim that guidance to match, but that is a decision to make deliberately, not a settled fact. Adding Mermaid diagrams is a low-cost win GitHub renders natively, and Han already uses Mermaid in several skill templates, so it extends an existing pattern rather than introducing one. A rendered docs site is a real option but carries a live risk (the most natural fit for Han, MkDocs + Material, entered maintenance mode in November 2025), so it is a later, separate call.

**Confidence: Medium.** The baseline practices are corroborated across independent primary sources. The escalation layer is genuinely unsettled in the evidence; two of the "apply to Han" judgments are syntheses across differing real-world examples rather than single-source best practices; and the content-division question above is an open decision this research surfaces but does not resolve.

## Research Results

### Documentation is built in two layers: how it is produced, and how it is organized

The evidence separates cleanly into a production/maintenance concern and an organization concern, and they answer different questions.

**Docs-as-code is the dominant production practice, and its main benefit is anti-rot.** Writing documentation with the same tools as code — plain-text markup in the repo, version-controlled, reviewed in pull requests, built in CI — is documented by two independent practitioner sources on both its definition and its benefits: shared workflow with developers, freshness of authorship, and the ability to gate merges on missing docs (A4, A5). The strongest evidence for *why* this matters is a peer-reviewed study of over 3,000 GitHub projects, which found that most contain at least one documented code reference (a function, file, or class) that no longer exists in the source, and diagnosed the root cause as a workflow gap: nothing in ordinary development signals that a code change has invalidated documentation (A17). That study's own proposed remedy is an automated scanner that detects surviving-but-invalid references, not a review-gate prescription; the "keep docs in the same repository under the same review gate" mitigation is the one most consistently cited by the practitioner sources (A5, A16), with A17 supplying the evidence that the problem it addresses is real and widespread rather than the specific fix. The practice has real costs, corroborated by two independently motivated critiques (one non-commercial, one from vendors of competing tools): Git is a poor fit for non-developer writers, conventions drift without enforcement, and reviewing raw markup without a rendered preview degrades review quality (A6, A7). The failure mode is adopting the tooling without the supporting process, not the concept itself (A6).

**For organization, "separate content by reader need" is the recurring principle.** The Diátaxis framework splits a documentation set into four types keyed to distinct reader needs — tutorials (learning), how-to guides (task), reference (facts), and explanation (understanding) — and argues that mixing these within one document degrades usability (A1). GitHub's own developer-documentation guidance endorses the same framework (A32). Real adopters report it helps both authoring and findability, with two important caveats corroborated across independent sources: applying it initially makes an existing doc set *look worse* because gaps become visible, and migration is a multi-year effort, not a quick reorg (A2). There is also an evidentiary gap: an independent practitioner could not find research validating that these *specific four* categories are the correct taxonomy, and the framework author's defense of the split is an assertion rather than a cited study (A3) `[single-source on the taxonomy's optimality]`. A complementary idea, "Every Page is Page One," argues each page must stand on its own because readers arrive by search or deep link, not by reading top-to-bottom (A10). That premise — readers scan and jump rather than read sequentially — is corroborated by independent usability and cognitive-science research showing most readers scan, and that concise, scannable, front-loaded writing measurably improves usability (A13, A14).

### GitHub recognizes a specific "front door" file set, and renders a lot without any build

**Community health files are a fixed, documented set with strict placement rules.** GitHub actively checks each public repo for README, LICENSE, CODE_OF_CONDUCT, CONTRIBUTING, SECURITY, SUPPORT, GOVERNANCE, FUNDING, and issue/PR templates, resolving each file by looking in `.github`, then the repo root, then `docs` (A19, A20). opensource.guide converges on the same set and adds the practical rule to link CONTRIBUTING and CODE_OF_CONDUCT from the README so they are discoverable (A21). This is well corroborated across independent primary sources.

**A folder-level README is a free landing page.** GitHub auto-surfaces a README when browsing any folder, not just the repo root (A37). It also auto-generates a clickable outline (table of contents) and heading anchors from markdown headings with no extra syntax, and rewrites relative links per branch — GitHub explicitly recommends relative over absolute links for in-repo references (A38). Since February 2022 it renders Mermaid diagrams natively wherever markdown renders (repos, issues, PRs, wikis), with the raw diagram source staying diffable in the file (A30, A31). Together these give a project navigable, diagram-capable, per-area documentation with zero tooling.

**No source names a threshold for when flat markdown must become a rendered site.** This was an explicit negative result: none of the fetched sources supplied a file-count or word-count rule for when to escalate (A24–A32 searched). What is well-evidenced is the tooling landscape once a project does escalate — covered in the Options below — including one fact that materially changes the calculus: Material for MkDocs, the most common markdown-first choice, entered maintenance mode on 2025-11-05 (critical and security fixes only for at least 12 months, no new features), with its maintainers redirecting new work to a successor tool, Zensical; the underlying MkDocs project has had no release since August 2024 (A25, the maintainer's own announcement; A26 restates it but is interested-party vendor content, so this rests on strong primary evidence rather than independent corroboration).

### Multi-package projects split on where docs live, but agree every project needs one front door

Two credible large open-source projects chose opposite structures. Spotify's monorepos (via the open-sourced `mkdocs-monorepo-plugin`) keep a `docs/` folder beside each package's code, merged into one site, explicitly to answer "who owns which documentation" using folder-based ownership like CODEOWNERS (A33) `[single-source for the Spotify-origin claim]`. Kubernetes instead uses one central documentation repository, entirely separate from the many component repos it documents (A34). Both are real and evidenced; they optimize for different things (per-package ownership vs. one coherent cross-project narrative), and neither claims the other is wrong. The one point of agreement across all sources is that every project needs a single discoverable entry point regardless of internal structure (A21).

### Han's current documentation: strong content, flat and centralized structure, only one README

Han already practices docs-as-code and already enforces a one-canonical-source rule, so the reorganization is a structural change, not a philosophy change.

- **The canonical-source convention is codified in two places.** CLAUDE.md and CONTRIBUTING.md both state: the long-form doc is canonical, indexes carry one-sentence scent plus a link, and the README never duplicates long-form content (A40). Any reorganization must preserve this, and issue #115 explicitly worries about breaking it.
- **All skills and agents have complete long-form docs, centralized under repo-root `docs/`.** Every skill has a doc under `docs/skills/{plugin}/{name}.md` and every agent under `docs/agents/han-core/{name}.md`; the indexes are complete (A41).
- **The two indexes are large.** `docs/skills/README.md` (162 lines) and `docs/agents/README.md` (99 lines) together hold every entry across nine plugin sections with internal sub-grouping — the "too large and complex" the issue names (A41).
- **Only the `han/` meta-plugin has a README.** The other plugin directories have none, despite `docs/plugin-readme.md` already carrying rules for what a per-plugin README should contain — including that skill directories must *not* have their own READMEs, and that a plugin README lists its skills with slash-command syntax (A42, A43). Importantly, that same draft guidance also specs a fuller "Skills Reference" section requiring, *per skill*, a paragraph description "more detailed than the one-liner," a Files line, and two to three example prompts — content that substantially overlaps what the long-form skill docs already carry (for example, `docs/skills/han-coding/investigate.md` already has its own example-prompts section). So Han's existing plugin-README guidance does **not** encode scent-plus-link only; part of it, if implemented as written, would duplicate long-form content. This is the unresolved content-division question, not a solved one (A43, A45).
- **Mermaid is already used in skill templates, and there is no site tooling.** Mermaid fenced blocks appear in at least three skill-reference templates — `han-coding/skills/code-overview/references/overview-template.md`, `han-core/skills/project-documentation/references/template.md`, and `han-reporting/skills/stakeholder-summary/references/stakeholder-summary-template.md` — plus a plan document, so the pattern exists as precedent rather than being absent. There is no `mkdocs.yml`, Sphinx `conf.py`, Docusaurus config, Read the Docs config, or docs-building CI workflow (A44).
- **A tenth plugin is arriving, and the docs have machine consumers.** A live sibling branch (`origin/han-communication`) adds a foundational `han-communication` plugin — depended on by every prose-producing plugin — and relocates the canonical readability and writing-voice references out of `han-core` into it; issue #115's own "Suspected Areas" already lists ten plugin directories, not nine. Any reorganization scoped off a nine-plugin count will be wrong by the time it ships, and the foundational plugin may warrant different README framing than a leaf plugin. Separately, Han's docs are consumed by Claude Code and referenced by internal tooling (the `han-update-documentation` skill's path-mapping table hardcodes `docs/skills/{plugin}/` patterns), so relocating docs or adding a plugin index has a machine-consumer blast radius beyond human GitHub browsing (A45).
- **The driving request.** Issue #115 asks for a README per plugin (what/how/why plus its components), slimmer alphabetized indexes that link out, a new "plugin index" pointing to the per-plugin READMEs, and *possibly* moving long-form docs into plugin-specific folders (marked as a maybe). The thread adds two live ideas from a contributor and the maintainer: rendering the docs via MkDocs/Sphinx on Read the Docs/Pages, and embedding Mermaid diagrams (A45).

## Options to Consider

These are grouped by the decision they belong to. The organization and front-door options (O1–O4) are the baseline; the rendered-site options (O5–O8) are the escalation layer.

### Baseline: organization and front door

**O1 — Per-plugin README as a scent-and-link "front door," central long-form docs unchanged.** Each plugin gets a short README describing what/how/why and listing its components as one-line scent plus links into the existing central long-form docs. The indexes slim to alphabetized link lists; a new plugin index points to the per-plugin READMEs. Steelman: matches GitHub's folder-README affordance (A37), gives every plugin a discoverable entry point (A21), and preserves Han's one-canonical-source rule because the README carries links, not duplicated content (A40). Directly delivers issue #115's structural asks. Trade-off, and a required decision: Han's existing draft plugin-README guidance (`docs/plugin-readme.md`) currently asks for a per-skill "Skills Reference" section that would duplicate long-form content (A43). Choosing O1 as written here means explicitly trimming that guidance to scent-plus-link; without that decision, O1 and the current guidance conflict, and the READMEs will drift toward duplicating the long-form docs. Rests on: A21, A37, A40, A43, A45.

**O2 — Also move long-form docs into per-plugin folders (the issue's "maybe").** In addition to O1, relocate `docs/skills/{plugin}/` and `docs/agents/` content into each plugin's own directory. Steelman: docs live beside the code they describe, enabling folder-based ownership, the pattern Spotify uses (A33). Trade-offs: larger, riskier migration, and a coupling the "two separate decisions" framing does not fully hold for — the Spotify/Backstage precedent achieves cross-folder discoverability *through* a rendered-site build (an MkDocs plugin that merges each folder's nav into one site), so under native GitHub markdown alone O2 would leave ten isolated plugin doc folders with no unified table of contents, unlike Kubernetes' central tree (A34), which is one browsable set with no build. The evidence is genuinely split, and no source measures which reduces onboarding friction. This option also carries the machine-consumer blast radius: the internal `han-update-documentation` path-mapping table would need updating. Rests on: A33, A34, A45.

**O3 — Keep the current flat-central structure, only slim the indexes.** Do the minimum: shrink the two indexes, skip per-plugin READMEs. Steelman: least effort, no migration risk. Trade-off: leaves nine plugins with no front door, contradicting the every-project-needs-an-entry-point principle (A21) and leaving the "wall of flat files" navigation problem the issue names. Rests on: A21, A45.

**O4 — Adopt Mermaid diagrams for flows and structure (independent of O1–O3).** Add diagrams to high-traffic docs (plugin relationships, dispatch flows, decision paths). Steelman: GitHub renders Mermaid natively with zero build, the source stays diffable in code review (A30, A31), scannable visual structure aids the nonlinear readers the usability research describes (A14), and Han already uses Mermaid in several skill templates, so this extends an existing pattern (A44). Trade-off: diagrams themselves can rot if not maintained under the same review gate — mitigated by docs-as-code (A17). Rests on: A14, A30, A31, A44, A45.

**Options considered and set aside.** A GitHub **Wiki** is a common answer to the "wall of flat files" problem, but it sits outside the pull-request review gate that the docs-as-code evidence identifies as the anti-rot mechanism (A6, A17), so it is a poor fit for a suite that already lives docs-as-code. **DITA / structured single-sourcing** (A15) solves multi-output single-sourcing that Han does not need, at a tooling cost far above its benefit here. **One consolidated README** fails the every-page-is-page-one and scannability findings (A10, A13) as content grows. One angle worth naming explicitly for Han: its docs are read by an agent (Claude Code) as much as by humans, which is an argument *for* the flat, greppable, relative-linked markdown the baseline keeps, and *against* a rendered site whose value (search, versioned nav) mostly serves human browsers — but no source in this research measures LLM-consumption fit, so this is reasoning, not evidence.

### Escalation: rendered documentation site (only if the baseline proves insufficient)

**O5 — Stay on native GitHub markdown, no site build.** Steelman: zero tooling overhead; Mermaid, auto-TOC, auto-anchors, relative links, and folder READMEs all work with no pipeline (A30, A31, A37, A38). Trade-off: no full-text search across docs and no versioned docs. Rests on: A30, A31, A37, A38.

**O6 — MkDocs + Material.** Steelman: fastest setup, markdown-first, simple YAML config, large install base (A24). Trade-off: entered maintenance mode 2025-11-05 with a 12-month critical-fix window and no new features; the maintainers themselves point migrators toward Zensical (A25). Choosing it today means planning a migration within roughly a year. Rests on: A24, A25 (A26 restates A25 but is interested-party, not independent).

**O7 — Docusaurus.** Steelman: built-in versioned docs, i18n, and Algolia-backed search that filters by version (A36); avoids the MkDocs maintenance risk. Trade-off: requires a Node/React toolchain and carries more maintenance overhead than MkDocs per its competitor's own comparison (A24) `[single-source on the overhead comparison]`. Rests on: A24, A36.

**O8 — Sphinx.** Steelman: best-in-class API-reference generation from Python docstrings via autodoc (A24). Trade-off: reStructuredText-first, steeper learning curve, and not evidenced as a strong fit outside Python/API-heavy projects — which Han is not. Rests on: A24.

Hosting, if any of O6–O8 are chosen, is its own sub-decision: GitHub Pages is free and simplest but has no native search, versioning, or PR previews (A28, A29); Read the Docs adds those and is free for open source, though the comparative claim is single-source from the vendor with only the "Pages lacks these" half independently confirmed (A27, A28); a custom host with PR previews (Kubernetes uses Netlify) is proven at scale but needs more setup (A34, A35).

## Recommendation

**Baseline (recommended, well-supported): O1 + O4, delivering issue #115's structural asks.**

Give every Han plugin a short README that states what the plugin does, how, and why, and lists its skills and agents as one-line scent plus links into the existing central long-form docs (O1). Slim the two indexes to alphabetized link lists and add a plugin index pointing to the per-plugin READMEs. Add Mermaid diagrams to the highest-traffic orientation docs (O4). This rests on corroborated evidence: GitHub surfaces folder-level READMEs as free landing pages (A37), every project needs one discoverable entry point per unit (A21), GitHub renders Mermaid natively at zero cost (A30, A31), and Han already uses Mermaid in its templates (A44). Han already lives docs-as-code, so this stays inside the review gate that keeps docs from rotting (A17).

The single most important design choice, and the one issue #115 itself flags as unresolved: **the per-plugin README should carry scent and links, not duplicated long-form content** — that is what keeps the reorganization from breaking Han's one-canonical-source rule (A40). But this is a decision to make, not a rule Han already enforces: Han's draft `docs/plugin-readme.md` currently specs a per-skill "Skills Reference" section (paragraph descriptions plus example prompts) that *would* duplicate the long-form docs (A43). Implementing O1 therefore means deliberately trimming that guidance to scent-plus-link before writing the plugin READMEs. The reorganization delivers issue #115's structural asks (a README per plugin, slim indexes, a plugin index), but it does not settle the content-division question issue #115 raises — that is an explicit open decision, not something this research resolves. Note also that the count is moving: the incoming `han-communication` plugin makes this ten plugins, not nine, and its foundational role may call for different README framing than a leaf plugin (A45).

**Deferred, no clear winner: whether to move long-form docs into plugin folders (O2 vs. O3's central structure), and whether to stand up a rendered site (O5–O8).** The evidence does not force either. On doc *location*, two credible large projects deliberately chose opposite structures (A33, A34) and no source measures the onboarding difference — so this is a deciding-criteria call, not a best practice: move docs into plugin folders if per-plugin ownership (CODEOWNERS-style) becomes a real need; keep them central if a single coherent narrative matters more. On a *rendered site*, there is no evidenced threshold for when flat markdown becomes insufficient (A24–A32, negative result), and the most natural fit for a markdown-first suite like Han — MkDocs + Material — carries a maintenance risk as of late 2025, on strong primary evidence (A25). Han is not Python/API-reference-heavy, so Sphinx's main advantage does not apply (A24). The defensible sequence is: do the native-markdown reorganization first (it satisfies the issue), then decide the site question separately, weighing search/versioning needs against the MkDocs maintenance clock, Docusaurus's heavier toolchain, or waiting for Zensical to mature.

**Evidence basis.** The baseline recommendation (O1 + O4) rests on corroborated evidence across independent primary sources (GitHub's own docs, opensource.guide, the GitHub Mermaid announcement, and Han's own codebase). The "front door carries scent not content" choice is grounded in Han's codified one-canonical-source rule (A40); it is a recommendation the research supports, not a rule Han already enforces at the plugin-README level, since the draft plugin-README guidance conflicts with it (A43). The deferral of O2 and the rendered-site decision rests on a genuine, documented split in the evidence (A33 vs. A34) and a negative result (no escalation threshold found in the sources searched), not on a single source — which is why they are presented as deciding criteria rather than a pick.

**What would change this recommendation:** an independent controlled study validating Diátaxis's specific four-category split would firm up the organization guidance; a documented threshold (file count, contributor onboarding time) for when flat markdown becomes unusable would let the rendered-site question be answered now rather than deferred; and Zensical reaching a stable release would remove the maintenance risk that currently weakens the MkDocs option.

## Validation

An adversarial validation pass challenged the evidence, the options framing, the recommendation, the Han-specific integrity, and the evidence-gathering integrity. The baseline recommendation (O1 + O4) survived, but several claims were corrected or qualified; the draft has been revised to reflect the findings below.

- **V1 (material, fixed).** The draft's claim that Mermaid appears "in exactly one plan document" was false. Mermaid fenced blocks exist in at least three skill-reference templates (`code-overview`, `project-documentation`, `stakeholder-summary`). This is a codebase-class error and was the strongest signal that the codebase evidence was less careful than its trust label implied. Corrected in Research Results and O4, which now frame Mermaid as extending an existing pattern.
- **V2 (material→near-fatal, fixed).** The draft claimed Han's `docs/plugin-readme.md` "already encodes" the scent-plus-link rule. It does not: its "Skills Reference" rule specs per-skill paragraph descriptions and example prompts that duplicate long-form content. This is the exact content-division tension issue #115 flags as open, and the draft over-claimed it as solved. The Summary, O1, and Recommendation now present scent-plus-link as a recommended decision requiring the guidance to be trimmed, and no longer claim the reorganization "fully satisfies" the issue.
- **V3 (material, fixed).** Source A26 (docsio.co), used to upgrade the MkDocs maintenance-mode claim to "well-corroborated," is vendor/answer-engine-optimized content, not independent editorial review — the same interested-party category the draft correctly flagged for A7. The registry now labels A26 as interested-party, and the MkDocs risk (in the deferred layer) is treated as resting primarily on A25, the maintainer's own primary announcement.
- **V6 (material, scoped to deferred O2, fixed).** The Spotify/Backstage precedent for moving docs into plugin folders (O2) achieves cross-folder discoverability through a rendered-site build, so "move docs into folders" is not cleanly separable from the escalation layer as the draft implied. O2 now carries this coupling caveat.
- **V7 (minor, fixed).** A17's peer-reviewed study proposes an automated scanner, not review-gating, as its remedy. The citation was narrowed to support the *problem* (widespread docs rot), with review-gating attributed to A5/A16.
- **V8 (material, fixed).** The codebase snapshot (nine plugins) is already stale: a live `origin/han-communication` branch adds a tenth, foundational plugin, and issue #115's own text lists ten plugin directories. Added to Research Results and the Recommendation, along with the machine-consumer blast radius (internal tooling references doc paths) surfaced by V5.
- **V4 (minor, fixed).** A miscitation (`A08`) was corrected, and several registered sources that did no evidentiary work in the body are acknowledged as background; the "corroborated across independent sources" framing was tightened to the sources that actually carry each claim.
- **V5 (material, fixed).** The options landscape omitted GitHub Wiki, DITA, and a one-README approach. An "Options considered and set aside" note now states each and why, plus the LLM-consumption angle relevant to Han.
- **V9 (minor, acknowledged).** The "no escalation threshold" negative result was independently spot-checked and held, but the spot-check was shallow; the claim is scoped to "not found in the sources searched," not "provably does not exist."

**What the validation did not overturn.** The core mechanism — every plugin (however many) gets a discoverable scent-and-link front-door README linking to unchanged central docs, indexes slim to link lists, a plugin index is added, and Mermaid is used where it aids orientation — was not refuted. Load-bearing external claims (A17's docs-rot finding, A21's community-file convergence, A33's Spotify framing, A25's MkDocs announcement) checked out against direct re-fetches.

**Confidence reasoning and remaining risks.** Confidence is **Medium**: the baseline is corroborated and Han already practices docs-as-code, but the content-division decision is unresolved, the codebase snapshot is moving under an incoming plugin, and the rendered-site layer is genuinely unsettled. Remaining risks the implementer must own: (1) decide explicitly whether to trim `docs/plugin-readme.md`'s "Skills Reference" section before writing plugin READMEs, or the no-duplication promise is not real; (2) recount for ten plugins and give `han-communication` README framing suited to a foundational plugin; (3) update the internal `han-update-documentation` path-mapping table (and any other machine consumers) when the plugin index lands or docs relocate.
